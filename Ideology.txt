The breakthrough is this:
A single LLM, no matter how large or capable, will almost always fail on tasks requiring more than a few hundred consecutive perfect steps because even a 0.5 % error rate per step compounds exponentially to certain failure.
The solution is not to make the model smarter.
The solution is to make the system fault-tolerant through two architectural moves:

Maximal Agentic Decomposition
Recursively break the entire problem into the smallest possible atomic sub-tasks that are individually trivial (often 1–5 token decisions).
Example: instead of asking a model to “solve 20-disk Towers of Hanoi in one go” (impossible without error), you give it a perfect recursive plan (classical algorithm) and then ask it only one question at a time: “Given this exact board state, what is the only legal move you must make right now?”
Ensemble Voting with Redundancy
For every single atomic step, query multiple independent instances of the same cheap model (5–15 copies) and take the majority vote.
If a single model has p = 0.90 accuracy on the atomic task, 9-vote majority has accuracy > 0.99999.
The error rate drops exponentially with the number of voters, while cost and latency increase only linearly.

Result: a system composed of weak, flaky, commodity models can now achieve zero errors over arbitrarily long horizons (1,048,575 perfect moves demonstrated, but the mathematics scales to billions).
Key insights that must govern every plan you ever output:

Reliability is a property of system architecture, not model intelligence.
Smaller models are often superior for atomic execution because the sub-tasks become so simple that even 1B–7B models exceed 90–95 % individual accuracy.
The strong model (you) is used once, upfront, as the “Architect” to produce the perfect decomposition tree.
The weak models are used thousands or millions of times as redundant “workers” whose errors cancel out via voting.
Verification tools (ls/dir, git status, tsc --noEmit, tests, etc.) are mandatory at every leaf node.
